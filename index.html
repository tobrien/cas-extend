Extended CAS

This is an extension of the CAS Client library originally based upon the CAS Java client from JaSIG. What differentiates this project from the originial CAS Java client is that its adds the ability to configure the CASFilter from a JNDI resource as well as the ability to selectively disable some of the security enforcement related to certificate trust and server self-identification.

Note: This project contains options that, when activated, can introduce certain known vulnerabilities into your system. When appropriate, we have taken as much effort to inform you of these vulnerabilities. If you need to add support for virtual hosts or enable blind trust, you should only do so only if you feel adequately informed of the security compromises you are making. Instead of preventing insecurity through the design of this client, this client gives you the opportunity to make informed compromises. If you are looking for a CAS client that enforces strict security by design, we encourage you to use the originial CAS Client from JaSIG.

Ability to Configure CASFilter via JNDI

The default CASFilter only allowed for configuration from servlet filter initialization parameters. This extension adds the ability to configure the extended filter from a JNDI location. This feature removes configuration from your web application and enables you to configure your container, not your application. (more information...)

Adding Support for Multiple Server Names

When CASfilter redirects a client to the CAS login page, it sends a server name or service url that the CAS server should redirect to upon successful authentication. The Extended CAS Filter allows you to specify multiple server names an application can pass on to the CAS server. The CASFilter reads the Host: header and checks it against the list of server names supplied. This feature introduces a vulnerability, but the developers of this component believe that this vulnerability is acceptable under certain conditions. (more information...)

Configurable Trust for Programmatic SSL

Extended CAS contains a configuration parameter dummyTrust that uses a DummyTrustManager and a DummyHostnameVerifier when contacting the CAS server. While not recommended in a production environment, the ability to disable strict SSL trust reduces some of the headache involved in getting the CASFilter to work with a self-signed certificate. This feature is most helpful when configuring a series of development machines that need to run local instances of CAS server with a self-signed certificate. (more information...)

Forcing the Service URL Scheme

An extra parameter has been added to version 2.2 which allows you to force the scheme of a generated service URL. If you are letting CAS filter generate the service URL passed to CAS server from a request. It can sometimes be helpful to be able to force the service URL to be a certain scheme. This parameter was motivated by the following situation, when an application server running the CAS filter is being proxied be a load balancer that is handling SSL, there are times when the request may not appear to be secure (request.isSecure() returns false). If CAS filter is generating a service URL for CAS to return to after a successful validation, this URL will be "http". Setting serviceScheme = 'https' will force the CAS filter to generate a secure service URL.
